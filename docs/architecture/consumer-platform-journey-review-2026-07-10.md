## 0. 总评

主规划的**数据建模是对的**：四层（工作台 → 报告发布 → 知识库 → Chat/研究）、不可变
publication version、关注≠成员≠入库三分离、三状态独立（已发布/归档中/可检索）、
官方走 feed 定制走 export/import——这些都无需推翻。

评审的核心担忧不在后端正确性，在**前端旅程的概念暴露**：一个后端必要的丰富模型，
被过度搬成了用户必须理解的导航与名词。若不收敛，风险是「架构优雅但用户觉得复杂」，
与此前推荐系统「技术正确但一眼看过去都是烂新闻」属同类失败模式。

对标结论本身写对了（`consumer-platform-product-design.md` §2.2）：AMiner 用户界面只有
Search / My Library / Chat 三个动词，丰富的多实体底座是**内部的**。以下修订建议的总纲
就是把这条结论落到本系统自己的界面上。

## 1. 修订建议（按严重度排序）

### R-J1（P0）用户可见概念上限：knowledge base / publication / route / snapshot 不上导航

- **问题**：主规划 §7.1 侧栏新增「统一搜索 / 知识库 / Chat / 研究」四入口，日报再叠
  「本工作台 / 已关注 / 发现」三 tab，叠加 publication / version / subscription / 主库 /
  topic 库 / route / context bundle / retrieval snapshot 一串名词。这与 AMiner「用户只见
  3 个动词」的对标结论自相矛盾——等于把内部模型搬成了用户导航。
- **必须改成什么**：在 `consumer-platform-product-design.md` 增加一节「用户可见概念上限」，
  显式声明——消费域首屏用户最多接触 **4 个动词：搜、问、存、研究**；
  knowledge base / publication / version / route / retrieval snapshot / context bundle
  是**内部概念，不作为导航对象或首屏名词出现**。所有面向用户的文案与 spec 服从此清单。
- **验收怎么查**：`frontend_control_governance.json` 的文案审计（copy_audit_rule）
  banned_terms 增补上述内部名词的**界面出现**为违例（代码/契约/审计 detail 豁免）；
  page-specs 每个消费域页在「首屏名词清单」栏列出实际暴露的用户名词，超过 4 个动词
  语义即评审不通过。

### R-J2（P0）主知识库隐形化：不做一等导航对象，术语对用户保留「文库/资料库」

- **问题**：主规划让每个工作台自动 provision 一个主知识库，并把 `/knowledge` 做成中心页。
  对单人/小团队用户，「工作台」已是他的容器；再让他学「知识库」这个词、去单独页面管理它，
  精神上就是主规划 §1 刚纠正掉的「双容器」换名字回来（工作台 + 知识库两个都可导航）。
- **必须改成什么**：
  1. 主知识库**默认隐形**——定位为「我的工作台产出自动变成可对话的记忆」。普通用户
     从不需要主动进 `/knowledge` 也能用 Chat；`/knowledge` 只在用户真的要建**隔离 topic 库**
     时才浮现（渐进披露）。
  2. 面向用户的术语保留现有**「文库 / 资料库」**，不把 `personal_libraries` 迁移后强行改叫
     更技术的「知识库」。ER 图与 API 用 `knowledge_base` 无所谓（主规划 §8.2 API 命名
     纪律照旧），但**用户界面词表**为用户服务。
- **验收怎么查**：viewer 首次进入、未建任何 topic 库时，导航**不出现**独立「知识库」入口，
  Chat 仍可用且默认对本工作台记忆提问（page-specs 补该空态旅程断言）；界面词表审计中
  「知识库」作为**用户可见标题**列入 R-J1 的复核面（内部 API/字段不受限）。

### R-J3（P0）Deep Research 范围声明常驻，防「期望落差」翻车

- **问题**：用户参照 AMiner「沉思」（含联网的多源深度研究），而本系统 v1 = 只检索站内
  已入库对象、不联网（`consumer-platform-product-design.md` §2.3、§3.3 已注明 intranet
  pull-only 边界）。若某工作台只有几天日报，研究产物会很单薄，用户会复现「功能弱」的
  失望——与「推荐很烂」同型。诚实标注写在了设计里，但**旅程入口没有把预期钉死**。
- **必须改成什么**：`/research` 发起框与全局研究侧板发起器旁**常驻一句范围声明**：
  「研究范围 = 本工作台已采集/已入库内容（不联网）」；信息缺口时明确引导「去数据源
  管理加源」，而不是让用户以为功能弱。planning/plan_review 阶段若检索面为空或过薄，
  必须显式提示「当前工作台可检索内容较少，建议先补充数据源」。
- **验收怎么查**：page-specs §32（研究页）与全局侧板 spec 断言范围声明文案常驻；
  空/薄语料下的引导态有测试；文案不得出现「全网/联网/实时抓取」等超出能力的措辞
  （纳入 R-J1 审计）。

### R-J4（P1）Chat 工作台切换语义：二选一定死，不用「restore before send」糊过去

- **问题**：主规划 §6.3 / `consumer-platform-product-design.md` §3.8 写「切换 AppShell 工作台后，
  发送前必须恢复会话所属工作台」。这是**代码正确性陈述，不是旅程答案**。真实场景：我在
  工作台 A 的 Chat，壳切到 B 想问 B，系统「恢复」成 A → 用户意外。
- **必须改成什么**：明确二选一并写进产品设计（评审推荐 (a)）：
  - **(a) 会话随工作台（推荐）**：切换工作台即切换可见会话列表；A 的会话在 B 视图下不出现，
    要继续 A 的会话需先切回 A。心智一致：「Chat 永远是当前工作台的 Chat」。
  - **(b) 会话保留 + 归属横幅**：切到 B 仍显示 A 的会话，但顶部常驻横幅「此会话属于工作台
    A，发送将以 A 的范围检索」，并提供「转到当前工作台新建会话」按钮。
  「发送前恢复」这种隐式回退**不允许**作为最终旅程。
- **验收怎么查**：page-specs Chat 页补切换工作台的显式旅程断言（列表变化或横幅出现），
  覆盖「切换后发送」的 scope 快照归属正确性。

### R-J5（P1）发现/已关注 tab 条件出现，单工作台部署下日报页保持单视图

- **问题**：主规划 §7.2 的「已关注 / 发现」两 tab 假设有多个工作台互相发布 internal_public
  报告。规划部实际大概率**单工作台**，两 tab 长期空着 = 首屏噪音；此消费发现层的价值只在
  **多团队 cloud 部署**成立。
- **必须改成什么**：日报/周报页三 tab **条件渲染**——当且仅当存在 >1 个「当前用户可发现的
  发布工作台」（或已有任一关注）时才显示「已关注/发现」；否则日报页维持干净单视图
  （「本工作台」不显示为 tab，直接是页面主体）。判定用一个轻量 `GET` 计数，不引入前端猜测。
- **验收怎么查**：page-specs 日报/周报页补「单工作台部署（discover 计数=0 且无关注）时不
  渲染 tab 组」断言；多工作台场景补三 tab 出现断言。

### R-J6（P1）先切一条薄 RAG 纵向切片，别让旗舰价值排在 6 层依赖之后

- **问题**：主规划 §11 依赖链 WP5-D→E→F→G→H→I 六层串行，RAG Chat（用户最想要的
  「问我的日报」）落在 WP5-G，前面压着 publication + 主库 + 物化 + 切片/索引。用户要等很久
  才摸到旗舰能力；且串行链任一环延期，后面全堵。
- **必须改成什么**：在 §11 之前**显式定义一条最薄纵向切片先行**——「就最新日报/选中条目用
  `consumer_context_bundles` 问一问（无持久索引）」，复用已落地的 C1-C3 基线，让用户在
  M3 完整 RAG 之前就有可用的「问」体验；实施计划把这条标为独立可交付、不阻塞于 WP5-F 索引。
  产品文案必须区分「临时问答（已可用）」与「持久知识检索（索引就绪后）」，不把 bundle
  冒充 RAG（`consumer-platform-product-design.md` §5 开头已有此纪律，切片先行不得违反）。
- **验收怎么查**：01-implementation-plan 增一个「WP5-D0 薄问答切片」条目，验收 = 用户
  在无 pgvector/索引的环境下能对最新日报发起带引用的临时问答；文案审计确认未出现
  「RAG/知识检索已就绪」的越界宣称。

### R-J7（P2）guest 消费体验的张力：记录，不强行解决

- **问题**：消费平台旗舰是 Chat，guest 是「消费者」，但 guest 不能 Chat
  （`consumer-platform-product-design.md` §3.4 判定正确：共享账号无个人域、预算不可归因）。
  公有云展示场景下访客的消费体验只剩搜索。
- **必须改成什么**：不推翻判定；但在产品设计**显式记录这个张力**并给旅程答案：guest 检索
  结果行对「问/存」按钮不渲染（与假控件规则一致，已写），并在 `/search` 顶部或结果区给
  一次**温和的注册引导**「注册后可对结果发起对话分析」，把 guest 从「消费者」自然导流到
  「注册用户」。不做置灰按钮、不做弹窗骚扰。
- **验收怎么查**：guest `/search` 旅程 spec 断言引导文案出现且非阻断；`/chat` 直访重定向
  `/search` 的既有断言保留。

## 2. 一致性核对（通过，无需改）

以下已核对与既有硬约束自洽，仅记录以备实施方复用：

- publication/knowledge/event 链路**不写** `adoption_status`、raw/news、公司 SQL
  （主规划 §12 断言 11）——与「编辑只改报告层」「公司 SQL 锁死」一致。
- 同步边界：publication 走官方通道投影，subscription / 阅读历史 / 知识库**永不进** sync feed
  （主规划 §9）——与 intranet pull-only、定制源 export/import 互通自洽。
- 幂等（content_hash 不变不新增 version、不重复通知/索引）、软删、引用快照占位——已覆盖。
