# AI情报官协作研发案例总结

## 1. 背景

AI情报官用于支撑规划部技术情报工作。系统从零开始重构，目标不是复制旧系统，而是把已有字段合同、日报 SQL 格式、旧信源资料和新的源治理能力整理成一个可维护的新工程。

旧系统曾经投入约 2 人月完成基础能力。蓝区重构阶段用约 10 人天完成了主链路重建，并补充了源治理、补采覆盖率、周报采信管理、历史归档和内网接入预留等能力。

## 2. 重构目标

重构时保留以下内容：

- 公司 SQL 字段和内网导入格式。
- 旧系统已验证的日报五段结构。
- 历史信源、旧报告和实体大事记中的事实资料。
- 规划部现有的日报、周报和采信工作习惯。

重构时不保留以下做法：

- 不直接运行旧系统。
- 不让旧库表结构成为新系统运行入口。
- 不把历史素材自动进入当前推荐和公司 SQL。
- 不把前端页面写成静态样板。

## 3. 研发流程

### 3.1 明确边界

先整理系统边界和字段合同：

- `docs/00-system-design.md` 记录系统目标、主链路和阶段边界。
- `config/contracts/*.json` 固化字段、SQL、工作台、同步和扩展点。
- `docs/legacy-system-spec.md` 记录旧系统事实，避免凭印象迁移。

### 3.2 建立主链路

主链路按以下顺序落地：

```text
data_sources
-> workspace_source_links
-> raw_items
-> news_items
-> dedupe_groups
-> recommendation_items
-> generated_news
-> daily_reports
-> company SQL export
```

这个顺序保证每一步都有持久化结果，也能从最终 SQL 追回原始来源。

### 3.3 用小步提交控制风险

工程按可验收的小步推进：

- 先搭建 FastAPI、Vue、数据库迁移和测试框架。
- 再实现登录、工作台、共享源池和抓取。
- 然后补标准化、去重、推荐、日报和 SQL 导出。
- 最后补周报、补采覆盖率、历史归档、质量归档和同步/审计页面。

每一步都配套测试、文档和字段合同，避免只完成页面或只完成后端。

### 3.4 外网开发和内网复用

外网环境负责快速开发和验证界面、接口、字段和 SQL。内网复用时，不要求前端重写登录体系。只要是 Vue 前端，可以通过一段路由守卫脚本接入公司 IDaaS：

1. 前端路由守卫读取内网统一认证回调中的 code。
2. 后端调用已有服务，把 code 换成工号、姓名和组织信息。
3. 系统按返回身份建立会话，并复用原有权限模型。

这套方式把“外网开发完成后内网复用”的成本降到很低，不需要在每个项目里重新实现一套公司登录。

## 4. 工程产出

当前工程已经形成：

- 后端服务、前端工作台、数据库迁移和配置合同。
- RSS、paper RSS、页面源和历史补采入口。
- raw、news、dedupe、recommendation、generated news、daily report、weekly report 主链路。
- 公司 SQL 导出和字段校验。
- 周报采信项管理。
- Tech Insight Loop 源治理和评分配置迁入。
- 历史报告、实体大事记、质量反馈和旧任务归档入口。
- 需求、任务、同步、审计页面。

## 5. 复用价值

这次重构沉淀了几类可复用做法：

- 字段合同先行：重要字段写进 `config/contracts`，代码和文档都围绕合同实现。
- 原始数据不覆盖：任何生成、编辑和导出都能追溯到 raw。
- 扩展点明确：来源、评分、导出、登录、同步和领域包都有独立入口。
- 内外网解耦：外网做产品和工程验证，内网通过 IDaaS adapter 接入。
- 测试和校验前置：日报、SQL、推荐、抓取和导入都有可重复运行的测试。

## 6. 后续改进

后续需要继续补：

- 生产部署配置和备份恢复演练。
- 公网到内网同步包的实质导出和导入。
- SQL 条目级追溯页面。
- 更完整的历史补采 provider。
- 周报正文自动生成。
- 覆盖率报告归档和 CI 门禁展示。
