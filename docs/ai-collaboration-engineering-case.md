# AI情报官设计驱动重构案例

## 1. 背景

AI情报官用于支撑规划部技术情报工作。重构前已有旧系统、历史信源、日报 SQL 格式和一批参考工具，但这些材料不能直接作为新系统运行入口。重构目标不是照搬旧实现，而是保留已验证的业务合同，重新建立一套可维护、可扩展、可测试的情报工程。

旧系统曾投入约 2 人月完成基础能力。蓝区重构阶段用约 10 人天完成主链路重建，并补充了源治理、补采覆盖率、周报采信管理、历史归档、多环境同步和内网接入预留等能力。

## 2. 设计先行

重构开始前先写设计材料，而不是直接写页面和接口。

第一层是总设计。`docs/00-system-design.md` 定义系统定位、主数据流、边界原则和阶段范围，明确 AI情报官不是单纯新闻站，也不能写死成 RSS + AI 日报。

第二层是 SDD。`docs/software-design-description.md` 把系统范围、架构、模块、时序、DFX、设计模式和测试设计整理成总装版，作为白盒评价和后续接手开发的主设计文档。

第三层是开发提纲。`docs/01-implementation-plan.md` 把设计拆成阶段，说明每一步交付什么、如何验收、什么时候可以进入下一步。

第四层是机器契约。`config/contracts/*.json` 固定字段、SQL 映射、工作台、同步策略和扩展点，避免只靠自然语言理解导致实现漂移。

这四层材料共同约束实现：自然语言文档说明“为什么这么做”，机器契约说明“字段和流程必须长什么样”，测试和校验脚本负责证明实现没有偏离。

## 3. 分阶段实现

主链路按可验收步骤推进：

```text
data_sources
-> workspace_source_links
-> raw_items
-> news_items
-> dedupe_groups
-> recommendation_items
-> generated_news
-> daily_reports
-> company SQL export
```

这个顺序有两个好处：

1. 每一步都有持久化结果，可以单独测试和排查。
2. 最终导出的 SQL 可以追回日报条目、生成稿、标准化新闻、原始 raw 和数据源。

实现过程中按“小步可验收”推进：

- 先搭建 FastAPI、Vue、数据库迁移和测试框架。
- 再实现登录、工作台、共享源池和抓取入库。
- 然后补标准化、去重、推荐、日报和 SQL 导出。
- 之后补周报、补采覆盖率、源治理融合、历史归档、质量归档、同步和审计。

每一步都尽量同时补齐数据库迁移、后端 schema、API、前端类型、页面、测试和文档，避免只完成一侧。

## 4. 约束机制

重构过程中最重要的约束不是“写得快”，而是保证关键合同不漂移。

### 4.1 字段合同

公司 SQL 字段、日报五段内容、分类、时间格式和导出范围写入 `config/contracts/news_sql_mapping.json`，并由 `scripts/validate_company_sql.py` 做专项校验。这样内网导入问题不会只靠人工记忆修复。

### 4.2 追溯合同

原始抓取数据必须进入 `raw_items.raw_payload_json`。后续的标准化、去重、推荐、生成稿和日报编辑都不能覆盖 raw。SQL 导出后再通过 trace API 回查来源链路。

### 4.3 分类合同

成品新闻的一级分类使用规划部既有 AI 十分类。Tech Insight Loop 的板块、源标签和专家路由只作为源治理和评分先验，不能写入公司 SQL category。

### 4.4 扩展合同

数据源、评分器、导出器、登录方式、同步包和领域包都以扩展点形式保留。新增能力优先增加 adapter、配置或 handler，而不是改主链路。

### 4.5 文档同步

`AGENTS.md` 和 `docs/README.md` 规定了修改同步规则：改字段同步 contract，改 SQL 同步映射和校验脚本，改登录同步 auth 文档，改同步策略时同步 sync contract 和设计文档。

## 5. 需求响应与维护

重构不是一次性交付。开发过程中根据实际使用问题做了多轮维护：

- 内网 SQL 导入曾暴露日期和字段格式风险，后续固定 0505 预览为字段校验基准，并把 SQL 标题、时间格式、五段结构和 HTML 清洗纳入校验。
- 日报和周报需要从“生成内容”转向“可管理采信项”，因此补了日报采信、周报板块、排序、编辑和发布。
- 抓取结果需要解释“为什么某天内容少”，因此补了抓取覆盖率、每源入窗/窗外/缺日期统计和失败原因。
- 信源扩展诉求引入 Tech Insight Loop 源治理，导入 386 条源记录，并区分可抓取入口和待补入口。
- 推荐质量诉求引入结构化评分器，把准入等级、噪声类型、拒绝原因和专家路由写入推荐结果。

这些维护都尽量以可复用能力落地，不做一次性脚本或页面补丁。

## 6. 重构成果

当前工程已经形成：

- 后端服务、前端工作台、数据库迁移和配置合同。
- RSS、paper RSS、页面源和历史补采入口。
- raw、news、dedupe、recommendation、generated news、daily report、weekly report 主链路。
- 公司 SQL 导出、字段校验和导出 trace。
- 周报采信项管理和板块化整理。
- Tech Insight Loop 源治理和评分配置迁入。
- 历史报告、实体大事记、质量反馈和旧任务归档入口。
- 需求、任务、同步、审计页面。
- CI 覆盖率报告、生产部署检查和同步包骨架。

## 7. 复用价值

这次重构沉淀了几类可复用做法：

- 先写设计和契约，再分步落地实现。
- 原始数据、标准化数据、推荐结果和报告层分开存储。
- 旧系统资料只作为事实来源，不直接决定新系统结构。
- 对内网强相关的 SQL、登录、同步做独立边界。
- 把“需求变化”整理成配置、扩展点、测试和文档，而不是散落在页面逻辑里。

外网开发和内网复用也形成了固定方法。只要前端是 Vue 应用，就可以通过一段路由守卫接入公司 IDaaS 回调 code；后端复用已有接口，把 code 换成工号、姓名和组织信息，再进入本系统的本地用户和权限模型。这样外网完成开发验证后，可以较低成本接入内网认证体系。

## 8. 后续演进

后续重点不是推倒重来，而是在现有边界内补完整度：

- 同步包导入侧补业务对象 apply handler 和冲突报告。
- 历史补采继续扩展到归档页分页、sitemap 深挖、论文 provider 和 CSV 导入。
- 周报在采信项管理基础上补自动正文生成。
- 生产环境补一次真实备份恢复演练记录。
- 增加硬件或半导体 domain pack 样例，证明系统可以扩展到 AI 以外的规划板块。
