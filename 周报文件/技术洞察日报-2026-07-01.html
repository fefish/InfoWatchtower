1、LangChain发布Deep Agents代码解释器，用WASM+QuickJS实现无沙箱运行不可信代码

智能体平台、协议与执行系统 LangChain 代码解释器 安全风险

📋 要点：2026年6月30日，LangChain发布Deep Agents代码解释器，用于安全运行Agent编写的不可信代码。该解释器采用WebAssembly（WASM）提供进程内执行隔离，使用QuickJS作为JavaScript引擎，编译到WASM中运行。能力隔离通过桥接机制实现：解释器默认无文件系统、网络等权限，仅通过显式桥接函数暴露有限能力（如调用子Agent）。持久暂停通过序列化WASM线性内存到LangGraph状态实现，可跨进程恢复。LangChain已开源quickjs-rs和langchain-quickjs两个包。

📌 总结：LangChain的代码解释器为Agent安全执行提供了轻量级替代方案，相比完整沙箱（如容器）更节省资源，适合编排型Agent。对于AI Agent平台开发者，这种WASM+快照模式可借鉴用于构建安全的Agent运行时。同时，该方案对提示注入防御有参考价值——通过限制Agent代码的能力边界，降低恶意输入的影响。

2、Hermes Agent上线MoA功能，用开源模型组合超越GPT-5.5

智能体平台、协议与执行系统 Hermes Agent MoA 性能提升

2、南科大与阿里提出SwiftCache：利用NVLink共享异构模型KV缓存，多轮对话提速69%

AI推理与服务加速 SwiftCache KV缓存共享 性能提升

📋 要点：2026年6月30日，机智流报道SwiftCache论文。SwiftCache针对多轮对话场景中KV缓存难题：GPU显存不足时，KV缓存被迫卸载到CPU内存或SSD，通过PCIe重载成为主要延迟瓶颈（当历史词元从5k增长到50k时，PCIe重载时间占比从73.2%上升至86.2%）。SwiftCache的设计包括：1）层流缓存：本地GPU只保留当前执行层的KV缓存，其他层缓存存储在其他GPU（Worker）显存，通过NVLink预加载。2）弹性缓存：Worker模型根据负载动态借出/归还显存，采用块优先布局实现O(1)调整。3）利用异构模型间空闲NVLink带宽。实验在4块H20 GPU上，与SGLang（HiCache）相比，P99 TTFT降低67-69%，与vLLM（LMCache）相比降低54-58%。对Worker模型的干扰小于9.7%。最大上下文长度扩展1.58-3.98倍。KV缓存传输开销低于总延迟的1.4%。

📌 总结：SwiftCache为AI推理服务提供了创新的KV缓存管理方案，尤其适用于长上下文多轮对话场景。其利用NVLink和异构模型共享资源的思想，可以推广到其他推理服务场景。对于核心网架构师，类似的内存和带宽协同优化思路可应用于网络功能虚拟化中的状态共享。同时，论文中块优先布局和弹性缓存机制对推理框架的KV缓存管理有直接参考价值。

3、SeKV: 分辨率自适应语义KV缓存，长上下文推理GPU内存降低53.3%

AI推理与服务加速 KV缓存 长上下文推理 成本下降

📋 要点：大语言模型的长上下文推理中，KV缓存成为主导性内存瓶颈。SeKV提出分辨率自适应的语义KV缓存，将上下文组织为熵引导的语义片段，存储在多级内存层次中而不丢弃信息。每个片段在GPU上保留轻量摘要向量用于粗粒度路由，在CPU上保留低秩SVD基础用于按需的Token级重建。训练一个缩放机制，在解码时选择性展开与查询相关的片段。SeKV保持基础LLM完全冻结，仅增加不到0.05%的可训练参数。在四个基准上，SeKV平均比最强语义压缩基线提升5.9%，而在128K上下文下GPU内存相比于完整KV缓存降低53.3%。代码已开源。

📌 总结：SeKV直接应对了LLM长上下文推理中内存爆炸的核心挑战。其“分辨率自适应”理念——按需恢复Token级细节而非全程保留——是一种高效的资源-精度权衡方案。对推理服务加速领域，该方案有可能使长上下文场景（如文档分析、代码库理解、对话历史）在有限GPU资源下变得更可行。对AI基础设施团队，GPU-CPU混合存储的思路也可以扩展到其他类似的内存密集型AI工作负载。开源代码有助于快速进行实际部署测试。
